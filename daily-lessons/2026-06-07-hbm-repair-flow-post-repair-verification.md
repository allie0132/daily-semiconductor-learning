## HBM Redundancy Architecture & Repair Resources

Each HBM DRAM die is fabricated with dedicated spare rows and spare columns per bank to enable post-manufacture repair. In HBM2E (JESD235C), the typical allocation is **2–4 spare rows per bank**; HBM3 (JESD238A) expands this to up to **8 spare rows per pseudo-channel bank group** to accommodate tighter lithography defect densities. Spare column resources (I/O repair) address DQ bit-line opens or shorts uncorrectable by ECC alone.
- **Repair CAM:** On-die Content Addressable Memory compares every incoming row or column address against stored repair entries in real time, redirecting accesses to spare resources with zero additional latency cycle in modern implementations.- **eFuse array:** Non-volatile one-time-programmable (OTP) cells store repair addresses permanently. After power-up, the eFuse array is sensed and loaded into the repair CAM latches.- **Repair address encoding:** A full repair address entry in HBM3 encodes {Die[2:0], Channel[2:0], Pseudo-Channel[0], Bank Group[1:0], Bank[1:0], Row Address RA[14:0]} — 27+ bits per entry. Address collision between two entries targeting the same spare row is a fatal programming error and must be checked algorithmically by ATE repair software before eFuse burn.

## Repair Classifications — Manufacturing, sPPR, and hPPR

Three distinct repair tiers exist in HBM production and test:
- **Manufacturing (Pre-Package) Repair:** Executed at wafer sort using laser fuse or electrical eFuse programming on the bare die. Addresses are determined from wafer-level MBIST. This is the primary repair pass for raw defect density; yields bonded known-good die (KGD) for 2.5D integration.- **sPPR — Soft Post-Package Repair (JESD235C §3.19.1):** Volatile repair held in flip-flops. Programmed via Mode Register Write (MRW) command sequence without eFuse burn. Cleared on power cycle. Primary use: final test screening, system-level test, and debug without consuming irreversible eFuse budget. An sPPR-passing device can be shipped with repair addresses stored externally (e.g., in system SPI flash) and re-applied at every boot by the memory controller PHY.- **hPPR — Hard Post-Package Repair (JESD235C §3.19.2):** Non-volatile; burns on-die eFuses permanently. One-time-programmable per entry slot. Used when post-package test uncovers defects not visible at wafer sort (e.g., stress-induced or assembly-induced fails). Requires power cycle after programming before the repair takes effect.JESD235C defines a maximum of **one hPPR operation per repair entry** (OTP constraint). Exhausting all spare rows plus all repair entries results in a hard bin failure.


## hPPR Command Sequence & ATE Implementation

The hPPR flow must be executed with precise command ordering and timing. The canonical sequence per JESD235C §3.19.2 is:
- **Step 1 — MBIST execution:** Run a full-coverage MBIST algorithm (e.g., March-C or GALPAT variant) via the ATE to capture all failing addresses {Die, PC, BG, BA, RA[14:0]}.- **Step 2 — Enter PPR mode:** Issue Mode Register Write (MRW) to the PPR-enable register. Specific MR address and bit field are vendor-defined within the JEDEC framework (commonly MR4 or a vendor extension register); consult die vendor datasheet. The PHY must assert CKE and hold the device in a quiescent state.- **Step 3 — Present failing address:** Issue an `ACTIVATE (ACT)` command with the failing row address RA[14:0] on the CA bus. Bank address is simultaneously presented.- **Step 4 — Latch and program:** Issue `PREAB` (Precharge All Banks). This command latches the repair address into the programming circuit and initiates eFuse blow. **Do not issue any other commands during programming.**- **Step 5 — Wait tPPR:** Hold quiescent for **tPPR ≥ 150 ms** (JESD235C minimum; vendor-typical implementation margin pushes this to 200–250 ms). Insufficient dwell time is the leading cause of marginal eFuse programming and field repair escapes.- **Step 6 — Exit PPR mode:** MRW to clear PPR-enable bit.- **Step 7 — Power cycle:** Mandatory for hPPR. The eFuse sense circuit re-loads the CAM only during the power-up initialization sequence (`tINIT`). Without a full power cycle, the repair CAM does not reflect the newly burned entry.For sPPR, steps 4–7 are replaced by a direct MRW to a repair address shadow register — no eFuse burn occurs, and the repair is active immediately without power cycle.


## Post-Repair Verification Strategy

Post-repair verification must confirm three independent properties: (1) the repair address was correctly stored, (2) the previously failing address no longer produces errors, and (3) the repair has adequate parametric margin to survive voltage and temperature excursions.
- **eFuse Read-Back:** After power cycle, issue Mode Register Read (MRR) to the die's repair status registers to read back the programmed address fields. Compare bit-for-bit against the intended repair address. Any mismatch indicates a failed eFuse cell (open-circuit blow failure or marginal threshold) and must result in immediate bin failure — do not re-program over a mismatched entry.- **MBIST Re-Run at Nominal Conditions:** Re-execute the same MBIST algorithm at nominal Vdd (1.2 V for HBM2E; 1.1 V for HBM3) and nominal temperature. Zero failing addresses at the previously repaired location is the pass criterion.- **Voltage Margin Sweep:** Repeat MBIST at **Vdd −5% and Vdd +10%** of nominal. Marginal eFuse programming can produce CAM match failures at Vdd corners even when nominal-voltage MBIST passes. This step exposes weak eFuse cells before shipment.- **Timing Parametric Re-Validation:** Spot-check `tRCD`, `tRAS`, and `tRC` at the repaired bank. Repair routing in some die implementations adds a small wire-length delta to spare row access paths; confirm timing margins have not collapsed below the JEDEC-specified minimums (e.g., tRCD_min = 14 ns in HBM2E at 2400 MT/s per JESD235C Table 86).- **Retention Bake (for hPPR only):** For qualification lots, a 150°C/24-hour bake followed by MBIST re-run verifies eFuse charge retention under accelerated thermal stress. Production test typically relies on characterization data and statistical sampling rather than 100% bake.

## Repair Yield Analysis, Budget Tracking & Escape Risk

Repair is a yield enhancement lever, not an unlimited safety net. ATE repair software must track and enforce hard limits:
- **Repair budget exhaustion:** Each die has a finite spare row count (e.g., 8 per PC per bank group in HBM3) and a finite eFuse entry count. If MBIST uncovers more unique failing rows than spare capacity, the die hard-bins. ATE repair algorithms must prioritize: prefer row repair over column repair where both are possible (row repairs are more resource-efficient per failing bit count), and pack multiple single-column fails onto one column repair entry if the JEDEC column repair granularity permits.- **Address collision detection:** Two hPPR entries targeting the same spare row index produce undefined behavior — one repair is silently lost. The ATE repair engine must run a collision check across all programmed entries before initiating any eFuse burn.- **Repair escape mechanism:** The dominant field failure mode from marginal hPPR is **eFuse delamination or charge redistribution under thermal cycling**, causing the CAM to miss a match on the repaired row. This manifests as intermittent ECC corrections escalating to uncorrectable errors (UE) in the field. Post-repair voltage margin sweep is the primary ATE-side screen; tighter tPPR dwell (250 ms vs. 150 ms minimum) provides additional guard band at modest test time cost.- **KGD gate for 2.5D integration:** HBM stacks entering an interposer assembly flow must complete manufacturing repair plus full post-repair verification before bonding. Re-test of the assembled 2.5D package via the PHY JTAG/BIST port provides a final assembly-induced defect screen, but physical access to individual HBM die MR commands is no longer possible; only stack-level commands through the PHY are available post-integration.

## Key Takeaways

- hPPR requires a mandatory power cycle after eFuse programming for the repair CAM to load; sFPR takes effect immediately but is lost at power-down — choose the repair tier deliberately based on test stage and downstream flow.
- tPPR dwell time (JESD235C minimum 150 ms; practical guard band 200–250 ms) is the single most critical ATE timing parameter for hPPR reliability; under-dwelled eFuse blows are the root cause of the majority of repair escapes.
- Post-repair verification requires eFuse MRR read-back, MBIST re-run at nominal conditions, voltage margin sweep at ±Vdd corners, and timing parametric spot-checks — nominal-only MBIST passing is insufficient to guarantee field reliability.

## Additional Learning: Parallel Pseudo-Channel PPR in HBM3 Cuts Repair Test Time

HBM3 (JESD238A) introduces independent PPR operation per pseudo-channel (PC0 and PC1 within each physical channel), allowing ATE to issue concurrent hPPR sequences to both pseudo-channels simultaneously rather than serializing them. Since tPPR dwell (≥150 ms) dominates the per-repair-entry test time budget, parallelizing across the two pseudo-channels reduces worst-case per-stack repair time by approximately 40–50% in high-repair scenarios (e.g., a device requiring 4 row repairs split evenly across PC0 and PC1). The ATE pattern generator must carefully interleave the MRW enable, ACT, and PREAB command streams on the two pseudo-channel CA buses without violating inter-command timing rules; most modern HBM ATE implementations handle this via a dedicated repair sequencer state machine in the pattern compiler rather than cycle-by-cycle hand-coded vectors.
